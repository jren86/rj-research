🔴 4. Codex CLI 致命 Bug：一年狂写 640TB，可报废 SSD

OpenAI Codex CLI 被曝 SQLite 日志 Bug——TRACE 级别日志写入忽略 RUST_LOG 环境变量：

  • 21天写入 37TB，年化 ~640TB——超过 1TB 消费级 SSD 全寿命保修写入量 (600 TBW)
  • SQLite WAL 模式引发写入放大——逻辑文件尺寸几乎不变，物理写入远超预期
  • 问题自 4月10日 GitHub Issue #17320 即已报告，10周无官方回应/补丁
  • 日志超 200MB 后 Codex 中途崩溃（30-105分钟）
  • ⚠️ 焊接 NVMe 笔记本最为危险——SSD 报废 = 整机报废

临时解决方案：Linux/macOS 用户可将 ~/.codex/logs_2.sqlite symlink 到 /tmp/（RAM-backed tmpfs）。Windows 无等效方案。

💡 Research Breakthroughs

💡 RL-Index：强化学习增强文档索引推理

RL-Index 将复杂检索推理从查询时转移到索引阶段：

  • 用 LLM 生成 rationale 增强文档索引，显式编码潜在查询-知识关系
  • GRPO + 检索相似度作为奖励信号优化 rationale 质量
  • 提升检索质量 + QA 性能 + 降低查询延迟
🔗 来源：arXiv TLDR

💡 DSG：解耦搜索与推理的 Vendor-Agnostic 架构

Decoupled Search Grounding (DSG) 将实时搜索与 LLM 推理解耦：

  • Vendor-agnostic：搜索供应商可自由路由/缓存
  • SimpleQA 接近原生准确率，但搜索成本降 91%，延迟降 68%
  • Agent 架构设计新范式——搜索不再绑定单一 LLM 供应商
🔗 来源：arXiv TLDR

💡 PARSE：企业文档 Prompt Injection 防御管线

PARSE (Provenance-Aware Retrieval Sanitization) 防御 LLM Agent 受 Prompt Injection：

  • 发现合成基准在真实企业文档上失效——现有方法全面低估风险
  • 分类句子 + 提取事实 + 一致性校验保留验证
  • 比 prior methods 更有效抵御真实企业文档中的 injection
🔗 来源：arXiv TLDR

💡 NRT-Bench：核电站控制室多轮红队评测

NRT-Bench 在模拟核电站控制室中对 LLM Agent 进行多轮红队攻击：

  • 5角色 LLM 操作团队 + 目标损害信号
  • 自适应多轮攻击可靠造成安全失效（8.7%-12.1% 关键功能丧失）
  • 安全关键系统中 LLM Agent 的脆弱性量化评估
🔗 来源：arXiv TLDR

💡 ContextRL：上下文感知 RL 提升多模态推理

ContextRL 用间接 RL 目标改善 LLM 在复杂上下文中的定位：

  • 从两个高度相似选项中奖励模型选择正确支撑上下文
  • 长程推理 +2.2%，多模态 VQA +1.8%
🔗 来源：arXiv TLDR
